fix(logs+filters): clean signature clustering, signature-aware actions, rerank union, placeholder drop

ROUND 5 — verified live against MBPStefano (192.168.1.153, macOS Monterey,
TeamViewer 15.71.4).

Five bugs surfaced by the previous live run, all fixed in this commit.

1) SIGNATURE NORMALIZATION (src/probes/logs.ts)
   The time regex /\d{1,2}:\d{2}(?::\d{2})?(?:[.,]\d+)?/ was eating the
   "99:15" prefix from macOS unified-log thread ids "[99:158b]", leaving
   "TeamViewer_Service[<time>8b]" in the signature. Same identical payload
   (e.g. 102x "curl request failed: Could not resolve hostname") was being
   split across 4 thread-id buckets (32/27/24/19) instead of clustered as
   one. Fix:
     * Strip the entire macOS unified-log compact prefix
       ("YYYY-MM-DD HH:MM:SS.fff  L   Process[pid:tid] [subsystem:cat] ")
       wholesale before any other replacement.
     * Defensive secondary strip for any Process[pid:hex_tid] tag that
       leaks through (e.g. text-log lines that prepend it).
     * Anchor the time regex on (^|\s) so it can never match colon-
       separated tokens that are not preceded by whitespace.
   Real-data effect: signatures are now the bare payload
     "HttpRequestImplCurl[<hex>]::CurlFinished(): curl request failed:
      Could not resolve hostname (6), Could not resolve host: router..."
   and clustering merges thread-id variants correctly.

2) SIGNATURE-AWARE ACTIONS (src/mastra/tools/specialistTools.ts)
   fromLogs() was emitting a single generic action ("Triage the dominant
   signature: search TeamViewer KB...") regardless of what the signature
   actually said. Replaced with a router that branches on the dominant-
   signature text and emits the right diagnostic playbook:
     * DNS resolution failures      -> scutil --dns, dscacheutil flush,
                                       dig @1.1.1.1, look for VPN/MDM DNS
                                       overrides
     * TLS / handshake failures     -> clock, CA bundle refresh, look for
                                       TLS-intercepting middleboxes
     * Transport instability        -> ping/MTR/tcpdump, NAT/firewall
       (timeouts, disconnects,         idle-timeout < 60s, rate-limiting,
       RetryHandle / RCommand,         Wi-Fi vs Ethernet
       NetWatchdog)
     * Auth class                   -> token scope/expiry, device
                                       enrollment
     * (default)                    -> previous "Triage" wording
   Bumped signature display slice 140 -> 200 and example slice 220 -> 260
   so the actual error wording stays visible after normalization.

3) RERANK UNION (src/mastra/workflows/teamviewerTroubleshootWorkflow.ts)
   The score-floor introduced in 44a4d67 only protected items the LLM
   chose to keep in its rerank reply. Small CPU models routinely return
   only a SUBSET of the candidates we asked them to rerank, silently
   dropping the rest. Fix: after the rerank, re-add every filteredRoots
   item that the LLM omitted, with score = original*0.6 (the same floor).
   Now evidence-grounded findings can no longer disappear because the
   model truncated its reply.

4) FIREWALL-ACTION FILTER REGRESSION (teamviewerTroubleshootWorkflow.ts)
   filterActionsAgainstEvidence had two over-broad heuristics that were
   dropping legitimate diagnostic actions:
     * "port5938Open && /\b5938\b/.test(text)" dropped ANY action that
       mentions 5938 -- including "capture tcpdump -p 5938".
     * "firewallRuledOut && /teamviewer|traffic|blocking|rules/" matched
       on the word "teamviewer" alone, which is in every TeamViewer-
       related action.
   Fix: introduce two carve-outs (probe-hygiene + diagnostic-context
   detection on tcpdump/pktmon/ping/traceroute/mtr/dig/nslookup/scutil/
   capture) and tighten the "blocking-recommendation" regex to actual
   block/allow/open-port/firewall-rule keywords. The existing "Check
   firewall settings for port 5938" still drops (no diagnostic verb,
   port 5938 proven open); the new "Transport-layer instability ...
   tcpdump on port 5938 ... check NAT/firewall idle-timeout" survives.

5) PLACEHOLDER DROP (teamviewerTroubleshootWorkflow.ts)
   The 1.5B-param Foundry-Local model sometimes emits literal "..." as a
   title/rationale/step (likely because its rendering aborted). After the
   rerank-union those placeholders no longer get filtered out by the
   score-based slicing. filterActionsAgainstEvidence and
   filterRootCausesAgainstEvidence now drop any candidate whose key text
   is < 6/10 chars or contains no 4+-letter word.
